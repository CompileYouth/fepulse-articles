## 设计师的工作没有消失，但工作的介质已经换了

Jenny Wen 这段分享最值得看的地方，不是她也在用 Claude，而是她把一件已经在 Anthropic 内部发生的变化说得很直白：**设计师不再只是画静态稿、写长规格、再把东西交给工程师实现，而是越来越直接地在 Claude Co-work 和 Claude Code 里把产品推向可运行状态。**

她描述自己的日常时，反复提到一个词：jam。也就是和工程师、产品经理围着一个正在运行的原型，一边看、一边用、一边改。这个过程比几年前松得多，也快得多。她现在不会花大量时间只盯住一个项目做完整设计稿，而是更像在五六个项目之间不断切换，给出方向、做判断、亲手修改一部分实现，再把下一轮变化交给工具和工程师继续推进。

这意味着，设计工作的核心并没有消失，但承载它的介质变了。以前设计往往停留在 Figma、规格文档和评审会上；现在很多设计判断已经直接发生在真实产品、真实内部 build、真实代码环境里。

## 规格还在，但已经从“厚文档”缩成了几条关键约束

Jenny 提到一个很典型的变化：团队当然还会做优先级排序，也还会写一些文档，尤其在需要交给安全、法务或发布相关团队时，还是得把事情说清楚。但这些文档已经不再是过去那种过度工程化、层层展开的大规格，而往往只是几条 bullet points。

Figma 也没有消失，但它不再是唯一中心。因为在今天这个工作流里，很多“原型”其实已经不是假原型，而是正在内部运行、能被真实体验的工作原型。设计师会先自己去用这个功能，形成对行为和体验的判断，然后直接坐到工程师旁边，说“这里该改、这里该收、这里还能怎么演化”。比起先做一份无比完整的设计稿再交付，这种方式显然更轻，也更贴近产品真实会发生的变化。

她讲得很清楚：现在更像是在代码里设计，而不是先设计、再翻译成代码。

## Co-work 最强的地方，不是帮你直接画界面，而是帮你从杂乱信息里提炼方向

Jenny 说自己现在很多事情都会先在 Claude Co-work 里做，而且她给了一个很好记的说法：**garbage in, treasure out**。意思不是输入垃圾就能自动变黄金，而是 Co-work 很擅长从大量零散来源里，把真正有价值的信号筛出来。她会把用户访谈、Slack 反馈、社交媒体讨论、评论区、Reddit 上的声音一起喂进去，让系统去总结现阶段最值得关心的洞察。

这里真正重要的，不是“又多了一个总结工具”，而是 Anthropic 的产品团队把它变成了离用户更近的一层操作台。她提到 Co-work 团队会尽量同时依赖传统的 UXR、内部 dogfooding、持续流动的 Slack 反馈，以及外部高热度用户的声音。然后再用 Co-work 去并行地处理这些信息，把其中值得迭代的部分提炼出来。

这其实已经很接近一个新的设计起点了。设计师不是先凭空构思一个界面，再慢慢补理由，而是先把真实输入吃透，再快速生成几个方向，继续往代码原型推进。

## 真正变化的不是工具，而是设计和工程之间的边界

对 FEPulse 读者来说，这段内容真正值得记住的是边界变化。Jenny 并没有说“以后设计师不用设计了”，相反，她多次强调 polish 和 iteration 仍然重要，评审也还在，很多判断还是需要人来做。但她同时也在展示另一件事：设计师现在已经能在 Claude Co-work 和 Claude Code 之间来回切换，既处理信息、形成方向，也真正动手把它推成工作原型。

这会让“设计师是否懂工程”“工程师是否参与设计”这些老问题，变得越来越没意义。因为当 prototype 本身就是可运行的，当产品讨论发生在真实界面和真实行为上时，设计和工程其实已经开始共用同一块工作台。以后真正稀缺的，可能不是谁更会写规格，也不是谁更会画稿，而是谁能在这种快速迭代里持续做出清晰判断，把杂乱信号收成产品方向，再把方向压进真实可用的体验。
