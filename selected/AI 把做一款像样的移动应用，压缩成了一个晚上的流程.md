## 一款过去要产品、设计、工程一起推进的应用，现在被压成了三段式流程

这条教程最有价值的地方，不是又做出了一个健身应用，而是把“从想法到成品”的流程压得非常清楚。讲解者一开始就把结论摆出来：过去要做出一个像样的移动应用，通常需要产品经理、设计师、工程师一起协作，但这次他是在下班后花几个小时，把一款能创建训练计划、记录组数、带休息计时、能在日历里回看训练历史的应用完整做了出来。

更重要的是，他没有把这件事包装成“Claude 一句提示词自动搞定”。他把整个过程拆成了三个阶段：先和 Claude 一起定义需求，再用 Pencil 做设计，最后用 Claude Code 加 Expo Go 完成构建和测试。也就是说，效率提升不是来自某个神奇按钮，而是来自一条更短、更连续、更少交接的工作流。

这条链路背后最关键的变化，是原来散落在产品、设计、开发之间的工作，现在被压进了一个人可控制的流程里。不是每个步骤都完全自动，而是每个步骤都能被 AI 加速到足够快，以至于个人也能把整条链走完。

## 第一步真正省下来的，不是写 PRD 的时间，而是把需求补全的来回成本

讲解者在第一步做的事非常典型：不是上来就让 Claude 写代码，而是先一起梳理需求。他先定义了几个核心页面，再明确告诉 Claude，“这只是基础，我们一起把 spec 做出来，如果你有问题，就用编号列表继续问”。这句要求很重要，因为它把 Claude 从一个被动执行器，变成了一个会帮你补需求空缺的“初级产品经理”。

后面发生的事也证明了这一步的价值。Claude 没有急着产出方案，而是连续追问：要不要自动 progressive overload、要不要内置休息计时器、日历如何交互、失败组怎么处理、重量单位如何切换、数据如何存储。很多需求并不是讲解者一开始就想清楚的，而是在 Claude 持续提问后逐步补全的。

这说明 AI 真正节省的，不只是文档时间，而是来回反复确认需求的成本。你以前要么自己一点点想到这些边角问题，要么依赖团队讨论把坑补上。现在只要你愿意让 Claude 先把问题暴露出来，需求完整度就会明显提高。

## 第二步最重要的，不是自动出图，而是把设计自由度从需求文档里拆出来

在转入 Pencil 之前，讲解者特地做了一件很关键的事：他让 Claude 把需求区分成用户可见需求和技术需求，并刻意不要在设计层面写得太死。他的理由很简单，下一步要交给 AI 设计工具去生成界面，如果前面的需求已经把视觉细节全部锁死，设计阶段就没有发挥空间。

这一步其实透露出一个很重要的工作方法。AI 工具串联起来之后，前一个阶段不应该把后一个阶段挤压成单纯执行。需求文档要足够明确，让设计工具知道要解决什么问题；但又不能详细到把设计空间全部抹平。讲解者在这里做的，不只是“把需求写出来”，而是在给后续 AI 设计保留弹性。

Pencil 的作用也因此被放大了。它不只是帮忙画草图，而是在几分钟里并行跑多个设计 Agent，快速产出完整的主流程界面、编辑界面、日历视图、训练历史页，再交由人继续调整。这和传统先出低保真、再细化、再评审的链路完全不是一个速度级别。

## 第三步真正改变开发体验的，是规格、设计和代码终于在一个晚上里接上了

教程里最后一步是 Claude Code 和 Expo Go。这里最值得看的，不是它真的把应用跑起来，而是规格说明、设计稿和真实代码之间的距离被压得很短。前面已经有了需求规格和设计稿，所以进入开发时，Claude Code 不需要从一团模糊想法里猜目标，而是能接住更清晰的输入。

这意味着过去常见的断层正在缩小。以前产品说需求、设计做稿、工程再翻译成代码，每一层都会丢信息。现在讲解者先和 Claude 把需求掰清楚，再让 Pencil 产出足够完整的设计，最后用 Claude Code 把它接进真实移动端环境里验证。整条链路里最慢的步骤，已经不再是“做出来”，而更像是“你知不知道自己到底想做什么”。

这也是这类教程真正让人不安、也让人兴奋的地方。它不是在证明 AI 能不能写几段界面代码，而是在证明一个过去需要多人协作、需要反复交接的流程，现在正在被压缩成个人可以独立推进的一套方法。**AI 改变的不是某个单点工具，而是把产品、设计、开发之间原本很长的链条，缩成了一条更连续的个人工作流。**
