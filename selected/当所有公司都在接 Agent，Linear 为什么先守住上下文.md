## 大家都在急着加聊天框时，Linear 先问了一句“这到底有用吗”

Karri Saarinen 回看 Linear 进入 AI 时代的过程，最值得记住的不是它做了多少 Agent 功能，而是它一开始并没有立刻冲上去。按照他的说法，Linear 的使命从来没变过，还是帮助团队把产品工作推进下去。AI 真正带来的变化，不是让这家公司突然换了方向，而是让“把负担从团队身上拿走”这件事终于有了更多可执行的方式。

也正因为如此，Linear 在最早一轮聊天机器人浪潮里并没有急着把自己包装成 AI 公司。Karri Saarinen 说，他们当时也试过把聊天式入口塞进产品，但很快发现这件事并不真的有用。问题不在于技术不能做，而在于没有清楚的工作流。你当然可以往产品里放一个聊天框，但它到底替谁解决了什么问题，这些才是决定价值的关键。

他把这种判断方式归结为一种设计背景带来的习惯：先理解，再动手。很多公司在技术上“能做”之后，会立刻进入“那我们现在就做”的状态，但 Linear 在意的是，这件事是否真的改善了团队工作的推进方式。

## 真正的变化不是把 SaaS 换成 Agent，而是把系统变成 Agent 的上下文层

Karri Saarinen 给出的判断非常关键：未来不会只有一个通用 Agent，而是每家公司、每个团队都会拥有很多不同的 Agent，甚至自己做出内部专用 Agent。那在这种前提下，Linear 的位置就不再只是任务管理工具，而会更像一个给 Agent 提供目标、上下文和决策边界的系统。

这也是为什么他反复强调，Linear 不想“拥有一切”。他们先做的不是闭门造自己的万能 Agent，而是先把平台层开放出来，把文档、接口和 Agent 集成能力做好。结果是，大量编码 Agent 很自然地就接入了 Linear，OpenAI 的产品也能直接把 Linear 当成外部上下文源来使用。客户也会自己做内部编码 Agent，再把这些 Agent 挂到 Linear 上。

这里最重要的一层变化是：Linear 的价值不再只是记录 issue，而是帮助组织决定“哪些事情值得被 Agent 执行”。Karri Saarinen 说得很直白，你可以让一百万个 Agent 同时动起来，但问题不在于能不能动，而在于它们到底该做什么。没有清晰的优先级、意图和背景，更多 Agent 只会制造更多无效劳动。

## AI 让执行更便宜之后，产品系统最稀缺的部分反而成了判断

这场对话真正击中产品和工程团队的地方，在于 Karri Saarinen 把 Agent 时代的问题重心重新摆了一遍。过去大家看任务管理工具，常常把它理解成一个记录流程、同步进度的地方。但如果 AI 越来越能替团队执行具体工作，那么系统最重要的作用就会变成提供判断框架：哪些需求是真的、哪些请求应该合并、哪些事情只是噪音、哪些才值得分配资源。

他举的一个方向是客户请求和反馈整合。Linear 现在重新加入聊天式入口，不是为了追赶风潮，而是因为他们终于知道这类入口在什么场景下能带来价值。比如客户问题本来就集中在 Linear 所掌握的上下文里，那 Agent 就能在这个基础上去归纳模式、抽取共性、帮助团队看到真正重要的问题。这样一来，聊天不是目的，而是把散乱信息压缩成可行动判断的手段。

这也是为什么 Karri Saarinen 会把 AI 时代的核心问题说成“如何更有生产力地 harness 这些能力”。Agent 已经能做很多事，但团队真正缺的不是执行按钮，而是决定什么该做、为什么要做、做到什么程度才算对。Linear 想抓住的，不是某个单点 Agent 功能，而是这层更稳定的意图系统。

## 如果 SaaS 没死 活下来的条件变了

这期标题用“如果 SaaS 已死，Linear 没收到 memo”做引子，其实不是在讲 SaaS 有没有死，而是在讲什么样的 SaaS 还能继续成立。Karri Saarinen 的答案很清楚：还能活下来的，不会是那种只提供一个静态界面、收纳任务列表的工具，而是能同时服务人和 Agent 的上下文基础设施。

这类产品的粘性，未来也不再来自“所有操作都必须在我这里完成”，而更可能来自“所有重要上下文都必须从我这里经过”。Agent 会越来越多，构建能力会越来越泛化，但真正稀缺的，是哪个系统掌握了组织的意图、优先级和反馈沉淀。Linear 想保住的，不只是界面，而是这层上下文主权。

对 FEPulse 读者来说，这场分享最值得记住的一句不是“大家都会有很多 Agent”，而是它背后的那层判断：**AI 越能执行，团队越需要一个能整理上下文、收束判断、定义意图的系统。** 真正没有收到 memo 的，不是 Linear，而是那些还把 AI 理解成“产品里多一个聊天框”的公司。
