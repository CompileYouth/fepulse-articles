## 把 Claude Code 当成主要设计工具，首先改变的是起手方式

这段分享最有意思的地方，不是再一次证明 Claude Code 能写页面，而是它把“做设计”这件事的起手方式改了。演示者已经连续一个多月把 Claude Code 当成自己的主要设计工具，用它做过定价页、对比表、评价区、常见问题区，也给一个金融看板做过营销首页。真正的变化在于，项目不再从 Figma 或现成组件库开始，而是从一个本地模板、一个空白页面和一段自然语言提示开始。页面结构、功能模块、版面层级，先用提示词长出来，再在浏览器里一点点调。

这意味着设计师不再只是把想法画出来再交给别人实现，而是直接在实现环境里试版式、试信息结构、试视觉判断。哪怕演示者承认自己对命令行还很不熟，能做的也只是切目录、启动 Claude，但这已经足够把设计工作推进下去。**新的门槛不再是先学完整前端，而是愿不愿意把设计工作搬进代码环境。**

## 好的 AI 设计，不是第一稿有多惊艳，而是第二轮开始能不能快速变准

演示里最清楚的一点是，Claude Code 的第一版并不是终稿，它只是一个还能继续雕的底稿。比如做金融应用首页时，先让它按“导航、首屏、功能区、数据区、评价区、行动区、页脚”这样的结构出第一版，然后立刻开始调。最先改的不是大方向，而是把首屏里那张“模型猜出来的假界面”，换成真实产品截图。因为只要页面里放进真实截图，哪怕没有额外视觉资产，首页也会马上从“像演示页”变成“像真实产品页面”。

接下来所有修改都很具体：把截图角半径收紧一点，让它跟页面其他圆角一致；把站点里的灰色从偏蓝的 slate 改成更中性的 neutral；把原来的实线边框换成灰黑色低透明度的外圈，让边框和阴影不再糊成一团。这里最值得记住的是，AI 生成的第一稿常常解决的是“有没有”，真正把页面拉到可用水准的，是后面这轮精细修正。

## 真正拉开质感差距的，往往是那些模型不会主动替你做的细节

这段演示一直在重复一个事实：AI 可以很快搭出框架，但它不会天然替你做审美判断。演示者之所以能把页面一步步拉顺，不是因为 Claude Code 自动知道什么是好设计，而是因为他不断把自己的判断灌进去。比如绿色虽然比常见的紫色更清爽，但整页最终还是决定回到更克制的深灰体系；比如 headline 虽然已经用了 Inter，但还要继续确认是不是 variable 版本，是不是启用了 display 版本，是不是该换掉带尾巴的小写字母，是不是该把字重从 500 和 600 之间调到 550，再把字距收紧一点。

这些动作看起来细，但它们决定了页面是不是还停留在“AI 做的那个味道”。Claude Code 擅长把结构搭起来，但真正决定页面高级不高级的，仍然是人对边框、阴影、字重、字距、对齐和配色的连续判断。**AI 没有取消设计感，反而把设计感变成了更显性的竞争力。**

## 从“默认居中”到“主动左对齐”，本质上是在夺回版式控制权

演示者还提到一个很典型的问题：AI 生成页面时，特别喜欢把东西都居中摆好，因为这是最容易做得整齐的默认解。但他自己的偏好是左对齐，所以会主动把这种默认排法扳回来。这看起来只是一个排版口味，背后其实是在说明一个更大的事实：Claude Code 可以非常快地产生“合格模板感”，但如果不去持续改，它就会停在一种正确却不够鲜明的平均值上。

所以这类工具真正适合的人，不是只会给一句大提示的人，而是愿意连续做几十个小判断的人。页面里的截图怎么放、颜色是不是太浑、边框该不该退到外圈、标题字重卡在哪个区间、版心应该居中还是左偏，这些都不是靠一次提示就能自动到位的。设计师的角色没有消失，而是从“亲手画每一个像素”，变成“不断校准系统默认值的人”。

## 当设计直接在代码里完成，设计师的工作边界就已经变了

这段内容真正指向的，不是“Claude Code 也能画页面”，而是设计和实现开始合并成一条连续链路。以前设计师交付的是静态稿，开发再把它翻译成网页；现在演示者直接在本地项目里改页面、抓截图、调字体、换边框、修颜色，设计稿和最终页面之间的距离被一下子压得很短。页面不是先画出来再实现，而是在实现过程中被设计出来。

这也是为什么这期内容对程序员和设计师都值得看。它讲的不是某个炫技案例，而是一个越来越现实的工作方式：**会用 AI 做界面的人，已经不再只是设计师。** 他同时在做信息架构、视觉判断、前端实现和产品包装。边界一旦松开，真正稀缺的就不再是“会不会用某个工具”，而是能不能把审美、结构和实现连成一条完整工作流。
