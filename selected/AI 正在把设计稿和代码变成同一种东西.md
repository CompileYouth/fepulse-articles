## 设计师最大的风险，可能不是被替代，而是还停留在旧工作流里

Felix Lee 开场就把问题说得很尖锐：很多设计师**还不够紧张**。在他看来，今天大量设计师的主工作环境依然停留在 Figma，真正进入 Claude Code、Cursor 这类编码代理工作流的人并不多。可一旦设计和实现之间的边界开始被打通，最危险的不是工具难学，而是人继续把自己限定在“只画界面、不碰产品”的旧角色里。Felix 自己是在 2025 年年底开始把“氛围式编码”真正用起来，几个月内就把这件事从尝试变成了日常工作方式。

他给出的建议也很务实：最适合起步的，不是接客户项目，也不是一上来做复杂商业产品，而是先做自己的个人网站。因为那是一个低风险、可反复试错、又和自己最相关的练手机会。先把一个小站从设计做到上线，人才能真正理解这套新工作流不是演示，而是可用的生产方式。

## Claude Code 的价值，不只是“能写代码”，而是产出质量足够高

Felix 对 Claude Code 的评价很明确：和他试过的其他编码代理相比，**从结果质量看，Claude Code 现在就是更强**。这不是单纯在比谁补全更快，而是在比谁更能产出接近可用产品的结果。他展示个人网站时，重点不是页面有多炫，而是一个设计师如何自己把深浅色模式、个人 AI 聊天助手、资料检索和真实数据接入都连起来。对他来说，这件事最关键的意义，是设计师第一次可以不经过传统前后端分工，就自己把想法直接推进到产品层。

他举的细节也很能说明问题。比如暗色模式图片，他并没有手工再做一张，而是让 Claude Code 直接把原图转成暗色版本。看起来只是小功能，但它背后代表的是：很多原本需要设计师和开发来回配合的细碎实现，现在被压缩进了同一条工作流里。**设计决策和实现决策，开始在同一个人手里闭环。**

## MCP 真正改变的，是设计稿和代码之间的数据形态

Felix 特别强调了 MCP 的价值，因为它解决的不是“能不能把界面截图贴进去”，而是“设计资产能不能继续保留可编辑性”。他提到，接入 MCP 之后，很多原本要手工补的设计信息会自动填充，而且生成出来的内容不是一张死截图，而是可以继续改颜色、改结构、改组件的矢量对象。也就是说，代码和设计稿之间不再只是互相参考，而是可以互相回流。

这也是他为什么觉得设计师不能只停在 Figma。以前设计稿交给开发，进入代码世界后就会变成另一套东西；现在通过代理和 MCP，设计稿、组件、代码和最终产品开始共享同一种更连续的数据形态。对设计师来说，这不是多学一个插件，而是要接受一个更大的变化：**界面不再只是“画出来”，而是“生成、修改、回流、继续生成”。**

## 真正拉开差距的，不是会不会写提示词，而是会不会给代理喂真实上下文

Felix 展示 ADPList 全球通话地图时，最有价值的部分不是视觉效果，而是他怎么把真实业务数据喂给代理。他把 AWS 后端导出的 CSV 数据、城市到国家的通话流向、会话时长和导师名单都交给 Claude Code，再给它一张 Shopify 地球仪的截图作为参考，让它先出计划，再开始生成。这个过程说明，代理最强的地方不是凭空造图，而是把**真实上下文、现成数据和参考界面**组合起来，快速长出一个可用产品。

更夸张的是，他并没有去买一个现成的 3D 地球素材，而是直接让模型自己生成。对他来说，这种工作方式最重要的不是省了多少外包成本，而是产品迭代速度被整体改写了。一个过去需要设计、前端、数据和可视化工程配合完成的东西，现在一个熟练使用代理的人，花十几个小时就能把雏形做出来。

## 设计和开发正在并成一条链，先适应的人会重新定义自己的职位

Felix 这场分享真正想推动的，不是让所有设计师都去当全栈工程师，而是提醒大家：职位边界已经在松动。设计师如果还把自己理解成“把稿子画好，交给别人实现”，很快就会发现自己离产品最有价值的那段链路越来越远。相反，谁能把设计、数据、代理和实现串到一起，谁就会离产品结果更近，也更难被替代。

所以这场演示最有冲击力的地方，不在于某个地球仪效果，或者某个聊天页面，而在于它把一个趋势说透了：**AI 正在把设计稿和代码变成同一种东西。** 当两边开始共用同一条工作流，真正值钱的就不再是单点技能，而是能不能从一个想法出发，直接把它推到上线。
