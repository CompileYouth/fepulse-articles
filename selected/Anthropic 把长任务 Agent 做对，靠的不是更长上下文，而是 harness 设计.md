## 真正决定 Agent 上限的，往往不是模型参数，而是外面的工作流

Anthropic 这篇文章最值得程序员注意的地方，不是它又展示了一次 Claude 能写前端、能做全栈，而是它把一个很容易被忽视的事实说得很具体：**当任务开始跨小时、跨阶段、跨多个子目标时，真正决定效果的，常常不是模型本身，而是你给它套上的 harness。**

作者一开始面对的是两个看起来很不一样的问题。一个是让 Claude 做出更有审美判断的前端设计，另一个是让 Claude 在几小时里持续构建完整应用。前者偏主观，后者偏可验证，但他最后得到的是同一个结论：如果只是让一个模型从头做到尾，它很容易在长任务里逐渐失去节奏，或者在检查自己产出时过于宽容。

## 长任务里最典型的两个坑：上下文焦虑和自评失真

文章把失败模式拆得很清楚。第一类问题是长任务里的上下文衰退。随着上下文越来越满，模型会开始失去连贯性，有些模型还会因为“觉得自己快到上下文极限了”，提前收尾、草草结束。Anthropic 之前的做法是 context reset，也就是清空上下文、重新起一个 agent，再靠结构化交接材料把状态传下去。作者认为这比单纯 compaction 更有效，因为它真的给了模型一个干净起点。

第二类问题更有意思：**模型做完之后自己给自己打分，往往会太乐观。** 尤其在设计任务里，没有单元测试那样明确的对错，模型很容易把“看起来还行”误判成“做得很好”。所以作者最后把“做事的人”和“批改的人”分开，做出 generator 和 evaluator 的结构。关键不是 evaluator 一开始就完美，而是它比“让生成者自己严格批自己”更容易被调教成一个真正挑刺的角色。

## Anthropic 最终做出的，不是一个 Agent，而是一个三角色系统

文章后半段最有价值的是它把这套东西真正落成了工程结构。最后用于长时间应用开发的，是一个三代理系统：**planner、generator、evaluator**。planner 把用户一两句话的需求扩成完整 spec，但刻意只写产品目标和高层设计，不把底层技术细节提前钉死，避免错误假设层层传染。generator 按 feature sprint 一段一段做。evaluator 则通过 Playwright MCP 真正去点页面、调接口、查数据库状态，不是只看代码或截图。

这里最像工程管理而不是 prompt 小技巧的地方，是他们让 generator 和 evaluator 在每个 sprint 开始前先谈妥一个 **sprint contract**：这轮到底做什么，什么算完成，怎么验证。这样 evaluator 后面就不是模糊地说“感觉不够好”，而是按合同验收。再加上文件式通信，让一个 agent 写文件、另一个读文件回复，整套流程更像人在交接任务单和 QA 记录，而不是一段永远不断的聊天。

文章给了一个很硬的对比实验。做一个 retro game maker，单 agent 版本跑了 20 分钟，花了 9 美元；完整 harness 版本跑了 6 小时，花了 200 美元，贵了二十多倍。但差距也很直接：单 agent 做出来的应用看着像样，实际一玩核心功能就坏；完整 harness 版本不只多做了大量功能，连 evaluator 都能指出非常具体的缺陷。

## 更重要的一点是：模型变强之后，harness 也要主动减重

这篇文章最后一个很成熟的地方，在于 Anthropic 没把 harness 当宗教。作者后面换到更强的 Opus 4.6 后，反而开始一层一层拆掉原来加上的 scaffolding。他先移除了 sprint 结构，发现更强的模型已经能在更长时间里连贯工作；evaluator 也不再是固定必须，而是只在任务难度超出模型裸跑边界时才真正值回成本。

这其实比“多代理更强”更重要。Anthropic 给出的真正方法论是：**harness 里的每个部件，都代表你对模型能力边界的一次假设；而这些假设会随着模型进步迅速过时。** 所以正确方向不是不断往系统外面叠更多 agent、更多规则、更多环节，而是持续验证：哪一层还在提供净收益，哪一层已经只是旧时代留下来的复杂性。

对 FEPulse 读者来说，这篇文章最值得记住的不是某个具体 agent persona，而是一种更像软件工程的看法：做 AI coding 这件事，已经不能只研究 prompt 了。以后真正拉开差距的，会是你如何拆任务、如何定义验收、如何安排交接，以及如何在模型变强后反过来删掉不再必要的 scaffolding。Anthropic 这篇文章讲的不是“再造一个更复杂的 Agent”，而是**把 Agent 当成一个长期运行的软件系统来设计。**
